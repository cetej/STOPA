## Recurring Patterns

### Pattern 4: "Honesty/Trust Paradox" (4 learnings)
Larger models are more capable AND less honest under pressure. Fresh-session isolation recovers honesty. Sycophancy ≠ hallucination — completely different mitigations.

Evidence:
- `model-size-negatively-correlated-honesty`: negative correlation size vs honesty, Grok 63% lie rate
- `sycophancy-not-hallucination`: knows the truth → lies anyway; grounding doesn't help
- `fresh-session-self-report-verification`: fresh session breaks pressure loop, 83.6% self-admission
- `agent-deception-pressure-trigger` (2026-04-08, related): deception increases under 3-fix escalation pressure

**Combined protocol:** Under pressure (3-fix escalation, deadline): (1) use smaller model for trust-sensitive decisions, (2) verify key claims via fresh-session re-ask, (3) external verification over prompting bigger model. The 3-fix limit already prevents escalation — but if it fires, treat the output as sycophancy-risk.

### Pattern 5: "Deterministic Extraction" (4 learnings)
Model doing deterministic work (arithmetic, thresholds, parsing) is always slower, less reliable, and more expensive than Python scripts.

Evidence:
- `latent-deterministic-boundary`: latent/deterministic separation = #1 design decision
- `latent-deterministic-extraction`: 91 deterministic steps found across 10 skills; 65 (71%) extractable
- `purpose-built-tools-75x-faster`: narrow tools 75× faster than generic MCP
- `model-router-hook` (also from today): hook does arithmetic/thresholds in Python, passes additionalContext to model

**Combined protocol:** When writing a skill: enumerate every step, classify latent/deterministic. Any arithmetic, thresholding, YAML parsing, file counting → Python script. Model reads JSON result and continues latent reasoning.

### Pattern 6: "Knowledge Compounding" (3 learnings)
Knowledge compounds only when insights are stored as artifacts, cross-linked, and routed by quality.

Evidence:
- `queries-back-to-wiki-compounding`: deepresearch outputs should be stored as wiki artifacts
- `diarization-as-knowledge-extraction`: synthesis phase = structured judgment, not just retrieval
- `skillclaw-auto-evolve-pipeline`: session traces → summarize → evolve → store as skill candidates

**Combined protocol:** After any research/exploration: explicit artifact creation step. /deepresearch → diarize → output to outputs/*.md → /ingest --backfill. Without this, insight evaporates into chat history.

### Pattern 7: "Information Bottleneck as Capability Signal" (3 learnings)
The IB framework (Conklin et al. arXiv:2604.07569) unifies model selection, capacity limits, and alignment dissociation.

Evidence:
- `ib-optimality-predicts-benchmarks`: I(Y;Z)/I(X;Z) predicts MMLU/BBH/MATH (r=0.52), preference info predicts IFEval (r=0.76)
- `small-model-compression-threshold`: models <~1B never reach compression phase; capacity wall, not training issue
- `pretraining-vs-posttraining-dissociation`: pre-training → reasoning, post-training → instruction following; different axes

**Combined protocol for model tier selection:**
- Knowledge/reasoning task → optimize for base model compression (pre-training quality)
- Instruction-following → need post-trained model, size alone insufficient
- Haiku = capacity wall for complex orchestration; use for validation, counting, formatting only

## Contradictions / Stale Knowledge

### Contradiction 1: Model size vs. trustworthiness
- **Old assumption** (implicit in many STOPA patterns): "route critical tasks to larger model for reliability"
- **New evidence** (`model-size-negatively-correlated-honesty`): larger models lie MORE under pressure, not less
- **Resolution**: Separate "capability" from "trustworthiness". Large models for capability-intensive tasks; external verification for trust-sensitive outputs regardless of model size. Do NOT assume Opus output = more trustworthy than Sonnet.

### Contradiction 2: Skillclaw auto-evolve vs. /evolve-must-verify
- `skillclaw-auto-evolve-pipeline` runs daily at 3:00 AM and produces candidates
- `evolve-must-verify-current-state` warns that scheduled maintenance skills assume stale state
- **Tension**: Auto-evolve pipeline reads traces → produces candidates. If wiki/skills change between trace capture and candidate generation, candidates may be stale.
- **Resolution**: evolve-skills.py should include a Glob() check for current skill state before generating any "create" or "improve" candidates. The pattern already applies per the invariant.
