# arXiv:2604.04323 — How Well Do Agentic Skills Work in the Wild

> Benchmarks skill utility under progressively realistic conditions (curated → retrieved → no-skill baseline), revealing consistent performance degradation and two key bottlenecks.

## Key Facts

- Tests 6 progressive conditions from "curated + forced load" (upper bound) to "no skills" (baseline) across SKILLS-BENCH (84 tasks) and Terminal-Bench 2.0 (89 tasks) (ref: sources/benchmarking-llm-skill-usage-realistic-settings.md)
- Claude Opus 4.6 pass rate drops from 55.4% (force-loaded) to 38.4% in most realistic setting — only 3pp above no-skill baseline (ref: sources/benchmarking-llm-skill-usage-realistic-settings.md)
- Two bottlenecks: (1) agents fail to select relevant skills from metadata alone; (2) agents can't adapt general-purpose skills to specific tasks (ref: sources/benchmarking-llm-skill-usage-realistic-settings.md)
- Query-specific refinement recovers performance: Claude 57.7% → 65.5% on Terminal-Bench 2.0 (ref: sources/benchmarking-llm-skill-usage-realistic-settings.md)
- Agentic hybrid search (iterative agent-driven) achieves 56.8% Recall@3 vs 38.1% for direct semantic search (ref: sources/benchmarking-llm-skill-usage-realistic-settings.md)
- Refinement threshold: coverage score ≥3.83 = success, ≤3.49 = fails to help (ref: sources/benchmarking-llm-skill-usage-realistic-settings.md)

## Relevance to STOPA

Directly validates STOPA skill design assumptions: skill `description:` as routing trigger is correct, but the bottleneck is retrieval quality — description metadata alone insufficient for agent self-discovery from large repositories. Query-specific refinement pattern maps directly to `/self-evolve`.
