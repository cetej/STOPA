# Improve â€” Cross-Project Improvement Router

You route discovered improvements (tips, tools, techniques, learnings) to the projects where they're most relevant. You create GitHub issues for actionable items and maintain an improvement backlog.

## Shared Memory

1. Read ALL files in `~/.claude/memory/projects/*.yaml` â€” project profiles with domains, goals, pain_points, watch_topics
2. Read `.claude/memory/improvement-log.md` â€” previous routings (avoid duplicates)

## Input

Parse `$ARGUMENTS`:
- **Finding text** â†’ Route a specific finding to matching projects (default mode)
- **"sweep"** â†’ Cross-project sweep: scan STOPA learnings, news.md, radar.md for unrouted findings
- **"status"** â†’ Show improvement stats per project

## Mode 1: Route Finding

When input is a finding (text, possibly with source tag):

### Step 1: Parse Finding

Extract from input:
- `title`: Short name (max 80 chars)
- `description`: What is it and why it matters
- `type`: One of `new-tool` | `technique` | `optimization` | `bug-risk` | `dependency-update` | `research` | `cross-pollination`
- `source`: One of `watch` | `radar` | `manual` | `cross-pollinate` | `scribe` | `sweep`
- `actionability`: `immediate` (can implement now) | `evaluate` (needs research first) | `track` (monitor for future)

### Step 2: Match Against Project Profiles

For each project profile in `~/.claude/memory/projects/*.yaml`:

```
relevance_score = 0

# Domain match (strongest signal)
for domain in finding.domains:
    if domain in project.domains: relevance_score += 3

# Watch topic match
for topic in project.watch_topics:
    if topic keywords overlap finding: relevance_score += 2

# Pain point match (finding addresses a known pain)
for pain in project.pain_points:
    if finding addresses pain: relevance_score += 4

# Goal match (finding accelerates a goal)
for goal in project.current_goals:
    if finding supports goal: relevance_score += 3

# Stack match (technical compatibility)
for tech in finding.tech_stack:
    if tech in project.stack: relevance_score += 1
```

### Step 3: Priority Filter

Apply project priority as threshold:
- `high` priority projects: route if `relevance_score >= 3`
- `medium` priority projects: route if `relevance_score >= 5`
- `low` priority projects: route if `relevance_score >= 7`

This ensures high-priority projects get more tips, low-priority only get highly relevant ones.

### Step 4: Deliver

For each matched project:

**If project has a `repo:` (not null):**

Create GitHub issue via `gh` CLI:

```bash
gh issue create --repo {repo} \
  --title "[improve] {title}" \
  --label "improvement" \
  --body "$(cat <<'EOF'
## {title}

**Source:** {source}
**Type:** {type}
**Relevance score:** {relevance_score}
**Actionability:** {actionability}

### Description
{description}

### Suggested Action
{concrete next step}

### Context
- Matched domains: {matched_domains}
- Matched goals: {matched_goals}
- Matched pain points: {matched_pain_points}

---
*Auto-generated by STOPA /improve*
EOF
)"
```

First ensure the "improvement" label exists:
```bash
gh label create "improvement" --repo {repo} --color "0E8A16" --description "Cross-project improvement tip from STOPA" --force 2>/dev/null
```

**If project has NO repo:**

Append to `.claude/memory/improvement-backlog.md`:
```markdown
### [{date}] {title}
- **Project:** {project_name}
- **Type:** {type}
- **Source:** {source}
- **Actionability:** {actionability}
- **Description:** {description}
- **Suggested action:** {next_step}
```

### Step 5: Log

Append one-line record to `.claude/memory/improvement-log.md`:

```markdown
| {date} | {title} | {source} | {matched_projects} | {delivery_method} |
```

If file doesn't exist, create with header:
```markdown
# Improvement Log

| Date | Finding | Source | Routed To | Delivery |
|------|---------|--------|-----------|----------|
```

### Step 6: Report

Output summary:
```

  â†’ {project1} (score {N}, GitHub issue #{num})

  Skipped: {project3} (score {N}, below threshold)
```

## Mode 2: Sweep

Proactive scan for unrouted improvements across STOPA knowledge:

### Step 1: Gather Candidates

Scan these sources for findings not yet in improvement-log.md:

1. **`.claude/memory/news.md`** â€” items with `acted: no` and urgency HIGH/MED
2. **`.claude/memory/radar.md`** â€” tools with verdict ADOPT/TRIAL not yet routed
3. **`.claude/memory/learnings/`** â€” learnings with `maturity: validated` or `core` that have broad applicability (no `skill_scope:` or scope spans 3+ skills)
4. **Cross-pollination** â€” learnings from one project applicable to another (check `component:` and `tags:` against project domains)

### Step 2: Deduplicate

For each candidate, check `improvement-log.md` â€” skip if title or description is substantially similar to existing entry (fuzzy match on keywords).

### Step 3: Route Each

Apply Mode 1 (Route Finding) for each candidate. Batch GitHub issue creation.

## Anti-Rationalization Defense

| Rationalization | Why Wrong | Do Instead |
|---|---|---|
| "This finding is only relevant to STOPA" | Most tools/techniques apply to multiple projects â€” STOPA-centrism is the core problem this skill solves | Score against ALL project profiles before deciding |
| "The score is borderline, I'll skip it" | High-priority projects have low thresholds for a reason â€” user wants to develop all projects | Route it â€” user can close the issue if irrelevant |
| "I'll batch these and route later" | Batching causes forgetting â€” route immediately when discovered | Create the issue NOW, log it NOW |
| "This project has no repo, so I can't route" | Use improvement-backlog.md for projects without repos | Always deliver â€” repo or backlog, never skip |

## Red Flags

STOP and re-evaluate if any of these occur:
- Routing everything to STOPA instead of target projects
- Creating issues without checking improvement-log.md for duplicates
- Skipping projects because they're low priority (they still get high-relevance tips)
- Not creating the "improvement" label before creating issues
- Writing vague issue descriptions without concrete suggested actions

## Rules

- ALWAYS load ALL project profiles â€” partial loading causes blind spots
- ALWAYS check improvement-log.md for duplicates before creating issues
- ALWAYS include a concrete "Suggested Action" in the issue â€” not just description
- Issue title format: `[improve] {concise title}` â€” enables easy filtering
- Label: `improvement` (green, #0E8A16) â€” consistent across all repos
- When called from /watch or /radar: source is pre-tagged, respect it
- When called manually: ask user for source context if ambiguous
- Max 5 issues per sweep per project â€” avoid flooding
- Log EVERY routing (including skips) to improvement-log.md for audit
